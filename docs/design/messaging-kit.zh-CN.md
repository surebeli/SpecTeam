# SpecTeam Messaging Kit

这份文档把当前定位沉淀成一组可以复用的对外文案模块。

它适用于 landing page、GitHub 描述、demo 讲解和短版产品介绍。

## 定位快照

### 市场侧命名

对外主名使用 `SpecTeam`。

产品、仓库和工作流对外统一使用 `SpecTeam`。

### 核心承诺

`SpecTeam 帮助团队发现 spec divergence、更快形成决策，并让人和 AI 工具保持对齐。`

### 扩展说明

`SpecTeam 帮助产品和工程团队发现 PRD、架构文档与 AI 生成方案之间的分歧，然后把一个清晰决策沉淀成共享事实来源。`

## 首页信息块

### Problem block

`你的 PRD 写了一套，架构文档写了另一套，Claude 提议 REST，Cursor 提议 GraphQL，最后没人知道哪个决策才算最终版本。`

### Solution block

`SpecTeam 会先找出分歧，再帮助团队形成决策，并把结果写回共享事实来源，让人和 AI 不再各走各路。`

### How it works block

1. 导入或编写 specs 与 proposals
2. 自动发现 divergence
3. 评审并形成决策
4. 把结果同步回团队的 shared source of truth

### Outcome block

- 更少彼此冲突的 spec
- 更快的方案收敛
- 更清晰的人机执行边界

## Demo Narrative

### 30 秒版本

`SpecTeam 面向那些 PRD、架构文档和 AI 生成方案经常彼此漂移的团队。它在一条 workflow 里先发现分歧，再帮助团队形成一个清晰决策，并把结果同步回共享事实来源。`

### 90 秒讲解结构

1. 从两份互相冲突的输入开始，比如一份 PRD 和一份架构方案。
2. 展示 SpecTeam 如何自动发现 divergence。
3. 打开一条具体分歧，演示 review 和 align 的过程。
4. 形成 decision 后，展示新的 source of truth 在哪里。
5. 最后强调价值不只是 review，而是让人和 AI 工具在重复决策里持续保持对齐。

### Demo payoff line

`重点不是再生成更多 proposal，而是停止让团队和 AI 工具基于不同假设工作。`

## 文案护栏

### 应优先强调

1. 自动发现 spec divergence
2. 更快收敛产品和架构决策
3. 让人和 AI 工具保持对齐

### 不应优先强调

1. 多 Agent 基础设施语言
2. MCP 实现细节
3. 企业部署包装
4. 产品尚未真正支持的完整 spec-to-execution 承诺

## 使用说明

1. 首句必须以问题为中心。
2. 尽量带一个真实矛盾示例。
3. 在用户理解楔子前，不要先讲大平台故事。
4. 在市场没有显著困惑或疲劳前，尽量复用同一套 headline 和 core promise。
