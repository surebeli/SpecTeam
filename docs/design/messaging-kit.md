# SpecTeam Messaging Kit

This document turns the current positioning into reusable copy blocks for the most visible external surfaces.

Use these blocks when writing landing pages, GitHub descriptions, demos, or short product intros.

## Positioning Snapshot

### Market-facing name

Use `SpecTeam` as the market-facing product name.

Use `SpecTeam` consistently across product, repository, and workflow surfaces.

### Core promise

`SpecTeam helps teams detect spec divergence, resolve decisions faster, and keep humans and AI tools aligned.`

### Expanded explanation

`SpecTeam helps product and engineering teams detect divergence across PRDs, architecture docs, and AI-generated proposals, then turn one decision into a shared source of truth.`

## Repo And Marketplace Copy

### GitHub About line

`Keep specs, decisions, and AI agents aligned.`

### GitHub short description

`AI-native spec review and decision alignment for product and engineering teams using multiple AI tools.`

### CLI package description

`CLI for SpecTeam spec review and decision alignment.`

### VS Code extension description

`Visual dashboard for spec divergence review and decision alignment in SpecTeam.`

## Homepage Messaging Blocks

### Problem block

`Your PRD says one thing. Your architecture doc says another. Claude proposes REST. Cursor proposes GraphQL. Nobody knows which decision is final.`

### Solution block

`SpecTeam finds the divergence, helps the team resolve it, and records the outcome so people and AI agents stop pulling in different directions.`

### How it works block

1. Import or write specs and proposals
2. Detect divergence automatically
3. Review and resolve decisions
4. Sync the result back into the team's shared source of truth

### Outcome block

- fewer contradictory specs
- faster design convergence
- clearer execution for both humans and AI tools

## Demo Narrative

### 30-second version

`SpecTeam is for teams whose PRDs, architecture docs, and AI-generated proposals keep drifting apart. In one workflow, it detects the divergence, helps the team make one clear decision, and syncs that outcome back into a shared source of truth.`

### 90-second walkthrough

1. Start with two conflicting inputs, such as a PRD and an architecture proposal.
2. Show that SpecTeam detects the divergence automatically.
3. Open one disagreement and walk through the review and alignment step.
4. Resolve the decision and show where the updated source of truth now lives.
5. Close by explaining that the value is not just review, but repeated decision alignment across humans and AI tools.

### Demo payoff line

`The point is not to generate more proposals. The point is to stop your team and your AI tools from working from different assumptions.`

## Messaging Guardrails

### Lead with

1. detect spec divergence
2. resolve product and architecture decisions faster
3. keep humans and AI tools aligned

### Do not lead with

1. multi-agent infrastructure language
2. MCP implementation details
3. enterprise deployment packaging
4. full spec-to-execution promises the product does not yet support

## Usage Notes

1. Keep the first sentence problem-led.
2. Use one real contradiction example whenever possible.
3. Avoid broad platform claims before the wedge is understood.
4. Reuse the same headline and core promise across surfaces until the market shows confusion or fatigue.
